Super Mario in C

Posted on 2013-04-09 Edited on 2022-01-25

This project implements a simplified version of Super Mario in C without any additional graphics library as a first year course project.

Turbo C (included in the repository in folder TC), as required by the course, and DOSBox are needed to compile and run this program.

Source Code: https://github.com/halfvim/SuperMario

Q&A

1. How to detect obstacles?

At first, we thought that since the background of the game could be pure blue, we could detect colors of the pixels around Mario to identify whether there is any obstacle. However, the behavior of Mario was weird, that it’s moving into the ground, as well as other obstacles. The approach we adopted made it hard for us to debug. After we researched a lot, we came up with a way that was reliable. We stored the map information in an plain text file with 0 representing background, different lower case characters representing obstacles, monsters, etc, and different upper case characters representing other background images like clouds, grasses, etc. Every character represents a 16 pixel * 16 pixel block in the game. The map has a fixed height and a very huge width, so that Mario could move horizontally. We stored both the current horizontal and vertical position of Mario, display corresponding images, and scan the obstacles according to the plain text file.

2. How to change the the display mode to video mode?

It seems very straightforward for some one who was satisfied by the 320 * 200 resolution provided by mode 13h. However, there still remains a question since we need 320 * 240 resolution so that the height could be dividable by 16. Using 16 * 20 block size is not ideal, since we found that the Mario and other elements are best displayed with square block size. We found some articles describing the video mode of Mode-X, but didn’t have any success using it.

3. How to play audio?

We used interrupt. Software interrupt instructions function similarly to subroutine calls and are used for a variety of purposes, such as to request services from low level system software such as device drivers. For example, computers often use software interrupt instructions to communicate with the disk controller to request data be read or written to the disk.[1] we used software interrupt instructions to send instructions to audio driver. The whole audio file was read into memory at first, and then the interrupt function reads the audio sequence and plays it at a certain rate. We actually found a piece of code on a book which could play a very small piece of audio. However, it’s very hard to adapt it to play bigger audio files. We tried a lot of times and managed to do it. We also looked into wav file structure and used some sound editing tool to generate wav file in certain formats from mp3 file.

4. How to display bmp image?

We needed to display a sequence of images at the beginning of the game. However, the images are displayed with incorrect colors. After some research, we found that the bmp file has color pallete in its header. The color palletes of the image files we had mismatched. So we edited the image files with photoshop, so that they could have the same color pallete.

5. How to use more memory than the 640KB limit?

Some document say that the memory limit of TC is 64 KB. However, we found that the actual memory limit in our practice is 640KB. But 640KB is still too small even to hold a single audio file. That’s why we used EMS. We used DOSBox to emulate DOS environment so that EMS is available.

6. Why the binary doesn’t run?

One of the possible reasons may be that you compiled your codes with wrong compiler mode in TC. Change it to large or huge, instead of small.
